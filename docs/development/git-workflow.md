# Git Workflow

## Overview

**Repository:** https://github.com/CBaney-IG/PurchaseSystem

This project uses **trunk-based development** with short-lived feature branches. `main` is always deployable.

```
main (always deployable → auto-deploys to production)
 ├── feature/USMP-42-approval-engine   (short-lived, merged via PR)
 ├── feature/USMP-01-initial-schema    (short-lived, merged via PR)
 └── hotfix/fix-email-token-expiry     (urgent, fast-tracked)
```

## Environments

| Environment | Branch/Trigger | URL | Supabase Project |
|---|---|---|---|
| **Production** | `main` (auto-deploy) | Vercel production URL | Production Supabase project |
| **Preview** | Any PR or branch push | Auto-generated by Vercel | Staging Supabase project |
| **Local dev** | Your machine | https://localhost:3003 | Local Supabase (`supabase start`) |

## Daily Workflow

### Starting work
```bash
# Use the Claude command (recommended):
/new-feature feature-name

# Or manually:
git checkout main
git pull origin main
git checkout -b feature/feature-name
```

### During work
- Commit frequently with clear messages
- Run `/validate` before considering work "done"
- If Claude goes off track, use `git diff` to review changes or `git stash` to set them aside

### Finishing work
```bash
# Use the Claude command (recommended):
/commit-feature

# Or manually:
npm run lint && npx tsc --noEmit && npm run test
git add -A
git commit -m "feat(scope): description"
git push origin feature/feature-name
# → Create PR in GitHub → Vercel creates preview deployment → Review → Merge
```

### Hotfixes
```bash
/hotfix brief-description
```

## Commit Message Format

```
type(scope): short description

Longer description if needed.

Refs: F-001
```

**Types:** feat, fix, docs, refactor, test, chore
**Scope:** area of code (auth, dashboard, db, api, etc.)

## Rollback

If something breaks in production:
```bash
# Option A: Revert the commit
git revert [commit-hash]
git push origin main

# Option B: Vercel instant rollback
# Go to Vercel dashboard → Deployments → click "..." on previous deployment → Promote to production
```

## Branch Protection (set up in GitHub)

Recommended settings for `main`:
- [ ] Require pull request before merging
- [ ] Require status checks to pass (Vercel build + tests)
- [ ] Do not allow force pushes
- [ ] Do not allow deletions
